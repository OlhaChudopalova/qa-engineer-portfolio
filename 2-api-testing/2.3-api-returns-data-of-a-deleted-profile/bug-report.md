# 🐞 Bug Report: Public API returns data of a deleted profile


**Priority:** High

## Environment
- Tool: Postman  
- Browser: Google Chrome 141.0.7390.122 (64-bit)  
- OS: Windows 11 Home 25H2  
- URL: stafferia.com

## Endpoints
- DELETE: `/api/v1/user/delete/`  
- GET: `/api/v1/executor/executor-profiles/{uuid}/`

## Description / Problem
Після видалення профіль користувача стає недоступним в каталозі спеціалістів на фронтенді.  
Проте через публічний GET-запит профіль залишається доступним і повертає більшість даних користувача (окрім контактів – згідно зі специфікацією), включно з прямим посиланням на незаблюрене фото.  
Така поведінка суперечить очікуваній логіці видалення, створює ризик витоку персональних даних та демонструє несинхронність доступів між frontend і backend.

## Preconditions
1. Користувач залогінений та має валідний access token.  
2. Відомий та зафіксований UUID користувача.

## Steps to Reproduce
1. Видалити профіль користувача через DELETE `/api/v1/user/delete/`.  
2. Переконатися, що API повертає 204 No Content. 
3. Перевірити, що профіль відсутній у каталозі спеціалістів на фронтенді та в адмін-панелі.  
4. Виконати GET-запит: `/api/v1/executor/executor-profiles/{uuid}` використовуючи UUID видаленого користувача.  
5. Проаналізувати відповідь.

## Expected Result
- Профіль користувача недоступний в каталозі спеціалістів на фронтенді.  
- GET-запит має повертати 404 Not Found або інший відповідний 4xx статус.

## Actual Result
- Профіль відсутній на фронтенді.  
- GET-запит повертає:  
  - `200 OK`  
  - статус профілю = `deleted`  
  - більшість даних профілю (окрім контактів – згідно зі специфікацією)  
  - пряме посилання на незаблюрене фото  

Це дозволяє ідентифікувати користувача та створює ризик витоку персональних даних.

## Security Impact
- Потенційний витік персональних даних.  
- Порушення очікуваної логіки видалення профілю.  
- Можливість несанкціонованого доступу до інформації через API.

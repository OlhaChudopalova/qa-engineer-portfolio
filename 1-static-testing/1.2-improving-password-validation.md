# Case: Improving Password Validation and UX

## Overview

This document presents the results of static testing and edge case analysis of the password validation logic on the Registration form.   

Based on the analysis, UX and validation improvements were proposed to make the password input more predictable, transparent, and user-friendly.  

The focus is on:  

- Validation logic consistency 
- UX clarity and predictability 
- Terminology correctness 
- Prevention of hidden usability traps 
- Alignment between system behavior and user expectations


---

## Context
Під час тестування форми реєстрації під полем Password відображаються вимоги до пароля, які допомогають користувачу коректно створити пароль.

## Original Requirements  

`Your password must meet contain:` 

> ✅ Not less than 8 symbols  
> ✅ 1 upper letter  
> ✅ 1 number  
> ✅ 1 special character (e.g., !, @, #)


## Original Validation Logic

- Кожна вимога позначається галочкою під час введення пароля, якщо вона виконується.  
- Система обмежує довжину пароля максимумом у 20 символів.

## Problem

Вимоги до пароля містять інформацію лише про мінімальну довжину, але не про верхню межу.  

Це створює UX-пастку:

- Користувач не знає про верхню межу.
- Введення 21-го символу блокується без пояснення.
- UI не дає зворотного зв’язку щодо перевищення довжини.


## Analysis

Прихована верхня межа довжини пароля створює непередбачувану поведінку інтерфейсу та підвищує когнітивне навантаження користувача.  
Користувач орієнтується лише на відображені вимоги; якщо система має додаткові обмеження, які не комунікуються, це формує приховану UX-пастку.  

Було проаналізовано два типи сценаріїв обмеження максимальної довжини:

### Upper limit (64–128 characters)
- Ризик перевищення ліміту є мінімальним.
- Користувачі рідко створюють паролі такої довжини.
- Явне відображення великого діапазону може створити психологічний дискомфорт (наприклад, користувач планує пароль з 10 символів, але бачить допустимі 128).

> Висновок: при такому діапазоні максимальну межу краще не вказувати явно.


### Upper limit (8–20 characters)
- Існує реальний ризик перевищення допустимої довжини.
- Користувач може досягти ліміту під час виконання останньої вимоги (наприклад, додавання спеціального символу).
- Блокування введення без пояснення виглядає як технічна помилка.

> Висновок: у цьому випадку явне відображення діапазону (8–20 characters) допомагає:
> - зробити поведінку інтерфейсу передбачуваною;
> - зменшити кількість непорозумінь;
> - уникнути фрустрації під час створення пароля.  

&nbsp;

## User Scenario (Edge Case)

**Hidden Upper Limit Scenario**

>  - Користувач вводить 8 або більше символів (наприклад, 20),
>  використовуючи комбінацію великих і малих літер та цифр.  
>  - Перша вимога “Not less than 8 symbols” позначається як виконана.  
>  - Інші вимоги (1 uppercase letter, 1 number) виконані.  
>  - Залишається лише остання вимога – спеціальний символ.  
>  - Спеціальний символ може бути складним для інтуїтивного підбору,
>  - оскільки вимоги не перераховують абсолютно всі допустимі символи.   
>  - Користувач намагається додати спеціальний символ як 21-й.  
>  - Система блокує введення через досягнення максимальної довжини (20 символів).


### User Impact

> - Користувач не розуміє причину блокування.
> - Може припустити, що спеціальний символ є некоректним.
> - Починає перебирати різні варіанти.
> - Витрачає додатковий час.
> - Виникає фрустрація через непрозору поведінку системи.

&nbsp;

# Improvement

- Явно відображати мінімальну та максимальну довжину пароля.

 Це забезпечує передбачувану поведінку інтерфейсу та зменшує фрустрацію користувача.

&nbsp;
## Static Testing / Microcopy Findings

Виправлення граматичних, орфографічних та термінологічних помилок у тексті вимог:
- “must meet contain” → “must contain” 
- “Not less then 8 symbols” → “Not less than 8 characters”
- “1 upper letter” → “1 uppercase letter”

&nbsp;
## Updated requirements:

`Your password must contain:`
> ✅  8–20 characters  
> ✅  1 uppercase letter  
> ✅  1 number  
> ✅  1 special character (e.g., !, @, #)

&nbsp;
## Outcome
- Користувач одразу розуміє допустимий діапазон довжини пароля.
- Неможливість подальшого введення більше не виглядає як “помилка”.
- Вимоги до пароля виправлені з точки зору граматики, термінології та орфографії.
- UX під час реєстрації стає передбачуваним та логічно послідовним.
